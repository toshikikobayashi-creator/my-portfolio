【1】システム概要
================================================================
「HUMAN AD セールスハブ」は、ヒューマンアド（LEDバックパック広告）と
アドトラック（LED搭載トラック広告）の営業DM送信を効率化する
オールインワンWebアプリケーションです。

3つの機能が1つのアプリで完結します：
  ① アタックリスト自動収集（企業リストを自動生成）
  ② AI DMジェネレーター（相手の情報に合わせてAIがDM文面を自動生成）
  ③ 営業管理ダッシュボード（ステータス管理・フォローアップ・CSV出力）

UIは全て日本語で作成すること。英語UIは不可。

--- 3-2. 実行SQL（Supabaseダッシュボードの SQL Editor に貼り付けて実行）---
以下のSQLを上から順にすべて実行すること：

-- テーブル作成
CREATE TABLE leads (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_name TEXT NOT NULL,
  contact_name TEXT,
  phone TEXT,
  email TEXT,
  website_url TEXT,
  facebook_url TEXT,
  category TEXT[] DEFAULT '{}',
  area_prefecture TEXT,
  area_city TEXT,
  priority TEXT DEFAULT '中',
  memo TEXT,
  source TEXT,
  status TEXT DEFAULT '未送信',
  first_sent_at TIMESTAMPTZ,
  follow1_sent_at TIMESTAMPTZ,
  follow2_sent_at TIMESTAMPTZ,
  follow3_sent_at TIMESTAMPTZ,
  next_follow_date DATE,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE dm_history (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  lead_id UUID REFERENCES leads(id) ON DELETE CASCADE,
  template_type TEXT,
  message_text TEXT,
  sent_at TIMESTAMPTZ DEFAULT now(),
  follow_number INT DEFAULT 0
);

-- updated_at自動更新トリガー
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER leads_updated_at
  BEFORE UPDATE ON leads
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();

CREATE TRIGGER dm_history_updated_at
  BEFORE UPDATE ON dm_history
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();

-- インデックス（検索・フィルター高速化）
CREATE INDEX idx_leads_status ON leads(status);
CREATE INDEX idx_leads_category ON leads USING GIN(category);
CREATE INDEX idx_leads_area ON leads(area_prefecture);
CREATE INDEX idx_leads_next_follow ON leads(next_follow_date);
CREATE INDEX idx_leads_company_name ON leads(company_name);
CREATE INDEX idx_dm_history_lead_id ON dm_history(lead_id);

-- RLS無効化（開発中）
ALTER TABLE leads DISABLE ROW LEVEL SECURITY;
ALTER TABLE dm_history DISABLE ROW LEVEL SECURITY;

--- 3-3. デプロイ時に追加で実行するSQL（Step 7で使用）---
ALTER TABLE leads ENABLE ROW LEVEL SECURITY;
CREATE POLICY "authenticated_access" ON leads FOR ALL
  TO authenticated USING (true) WITH CHECK (true);
ALTER TABLE dm_history ENABLE ROW LEVEL SECURITY;
CREATE POLICY "authenticated_access" ON dm_history FOR ALL
  TO authenticated USING (true) WITH CHECK (true);

【4】アタックリスト自動収集
================================================================

--- 4-1. 収集ソースと優先順位 ---
優先度1: Google Maps API（Places API New - Text Search）
  - Text Search APIを使用（旧Nearby Searchではなく新APIを使う）
  - 収集項目: 店舗名、住所、電話番号、Webサイト、評価、カテゴリ
  - 無料枠: 月$200分
  - Pythonライブラリ: googlemaps（Text Search対応）

優先度2: Webスクレイピング
  - Google検索結果からの企業情報収集
  - Playwright使用（既にMCP設定済み、Windows環境では cmd /c ラッパー必要）
  - レート制限に注意（リクエスト間隔3〜5秒）

優先度3: 公開企業データベース/API
  - 法人番号API（国税庁）: 無料、法人名・所在地
  - その他公開情報

優先度4: CSVインポート
  - 既存のAdTruck Lead Finder 300社データを取り込み
  - 手動で追加したリストの一括インポート

--- 4-2. Google Maps 全国検索の戦略 ---
Text Search APIで「{カテゴリキーワード} {都市名}」を検索する。

検索起点リスト（優先順）：
  東京（新宿/渋谷/銀座/池袋/品川）, 大阪（梅田/難波/天王寺）,
  横浜, 名古屋, 札幌, 福岡, 神戸, 京都, 仙台, 広島,
  川崎, さいたま, 千葉, 北九州, 堺, 新潟, 浜松, 静岡, 岡山, 熊本

検索の実行順序：
1. まず東京5エリア × 全カテゴリ（最優先）
2. 次に大阪3エリア × 全カテゴリ
3. その後、残りの都市を上記リスト順に
4. 途中で500件を超えたら一旦停止して俊輝に報告

--- 4-3. 検索カテゴリ・キーワード ---
カテゴリ1「イベント企画会社」:
  キーワード: イベント企画 / 展示会運営 / フェスティバル主催 / セミナー運営 / 催事企画

カテゴリ2「飲食・小売チェーン」:
  キーワード: 新規出店 / グランドオープン / キャンペーン / フランチャイズ / 飲食チェーン

カテゴリ3「広告代理店・PR・マーケ会社」:
  キーワード: 広告代理店 / PR会社 / マーケティング会社 / ブランディング / プロモーション

カテゴリ4「店舗型ビジネス」:
  キーワード: 不動産会社 / フィットネスジム / 美容サロン / クリニック / 学習塾

カテゴリ5「その他（広告予算がある企業）」:
  キーワード: IT企業 / アプリ運営 / ECサイト / エンタメ / レジャー

エリア: 全国対応（都道府県・市区町村で絞り込み可能にする）
目標件数: 初回500件以上

--- 4-4. FacebookページURL自動収集 ---
企業データ収集後、以下の追加ステップでFacebookページURLを取得する：
1. 収集した各企業について、Googleで「{会社名} Facebook」を検索
2. 検索結果からfacebook.comのURLを抽出
3. facebook_urlカラムに保存
4. 見つからなかった場合はNULL（空欄）のまま
5. 精度は約80%。見つからなかったものは手動で後から入力可能
※ レート制限に注意: 検索間隔は5〜10秒空ける

--- 4-5. 重複排除ルール ---
- 第1判定: 電話番号が完全一致 → 重複と判定
- 第2判定: 会社名の類似度が80%以上 → 重複候補フラグ
  （Pythonの標準ライブラリ difflib.SequenceMatcher を使用）
  （「株式会社」「(株)」「有限会社」等を事前に除去してから比較）
- 重複時は情報が多い方を残し、他ソースの情報をマージ

--- 4-6. 既存データ統合 ---
- AdTruck Lead Finderの300社プールをCSVインポートで統合
- CSVの場所: まずmy-portfolioリポジトリ内と ~/Downloads を検索
  → 見つからない場合は俊輝に「CSVファイルの場所を教えてください」と聞く
  → それでも見つからなければスキップして新規収集を優先
- インポート時に自動重複チェック
- カテゴリ「アドトラック」として自動タグ付け
- CSVインポート機能自体はUIに組み込む（後から手動でもインポートできるように）

--- 4-6-2. CSVインポートのフォーマットルール ---
CSVの列名はバラバラな場合があるため、以下のマッピングルールで自動判定する：

会社名 → company_name（列名候補: 会社名, 企業名, company_name, company, 名称, 法人名）
担当者 → contact_name（列名候補: 担当者, 担当者名, contact, name, 氏名）
電話番号 → phone（列名候補: 電話, 電話番号, TEL, tel, phone）
メール → email（列名候補: メール, メールアドレス, email, mail, Email）
サイト → website_url（列名候補: URL, サイト, HP, ホームページ, website, url）
Facebook → facebook_url（列名候補: Facebook, FB, facebook, fb_url）
業種 → category（列名候補: 業種, カテゴリ, category, 分類）
都道府県 → area_prefecture（列名候補: 都道府県, 県, prefecture, 所在地）
市区町村 → area_city（列名候補: 市区町村, 市, city, 住所）

マッピングルール:
- CSVアップロード後、まず列名を自動判定してプレビューを表示
- 「この列 → このフィールド」の対応表をユーザーに見せる
- 間違っていたらドロップダウンで手動修正できるUI
- 「インポート実行」ボタンで確定
- 文字コード: UTF-8とShift_JIS両方に対応する（日本語CSVはShift_JISが多い）

--- 4-7. エラー処理・リトライ ---
- API制限やネットワークエラー時、途中まで収集したデータはDBに保存

【5】AI DMジェネレーター
================================================================

--- 5-1. 仕組み ---
固定テンプレートではなく、Anthropic Claude APIを使って
相手の企業情報に合わせたDMを自動生成する。

使用モデル: claude-sonnet-4-20250514
API呼び出し: Next.jsのAPI Route（/api/generate-dm）からサーバーサイドで実行
※ APIキーはサーバーサイドのみで使用（クライアントに露出しない）

--- 5-2. APIレート制限 ---
/api/generate-dm にレート制限を実装すること：
- 1ユーザーあたり: 1分間に最大10回まで
- 実装方法: メモリ内のMap（IPまたはセッション単位）で簡易管理
  （本格的なものは後からRedis等で強化可能）
- 制限超過時のレスポンス: 429 Too Many Requests
  UIに「少し待ってから再度お試しください」と日本語で表示

--- 5-3. DM生成フロー ---
1. リスト画面から相手を選択
2. 以下の情報が自動で入力される：
   - 会社名、担当者名、業種カテゴリ、エリア
3. ユーザーがDMの種類を選択：
   種別A: ヒューマンアド提案（LEDバックパック広告の提案）
   種別B: アドトラック提案（LED搭載トラック広告の提案）
   種別C: セット提案（両サービス組み合わせ）
   種別D: フォロー1回目（軽いリマインド）
   種別E: フォロー2回目（価値提供型）
   種別F: フォロー3回目（ラストチャンス型）
4. 追加の指示欄（任意）：
   - 「イベント集客に焦点を当てて」「新規出店の話題に触れて」等
5. 候補日時の入力欄：3つ（手動入力、将来的にGoogleカレンダー自動取得）
6. 「AIで生成」ボタン → ローディング表示 → Claude APIがDM文面を生成
7. 生成されたDMをプレビュー → 自由に編集可能
8. 「コピー」ボタンでクリップボードにコピー
9. FacebookページリンクからDM画面に遷移して貼り付け・手動送信

--- 5-4. AIへのプロンプト（API Routeに組み込む）---
以下のシステムプロンプトをClaude APIに送る：

"""
あなたは株式会社HUMAN ADの営業担当です。
Facebook DMで企業にアプローチするためのメッセージを作成してください。

【HUMAN ADのサービス】
・ヒューマンアド: LEDバックパック（Jiatean JTA-WT32-BA）を背負ったウォーカーが
  街中を歩く移動型広告サービス。歩行者への直接的なアプローチが可能。
・アドトラック: 大型LEDビジョンを搭載したトラックによる走行広告サービス。
  動画・静止画で圧倒的なインパクトのあるPRが可能。走行ルート・スケジュールは柔軟にカスタマイズ可能。

【メッセージルール】
・350〜550文字で作成
・冒頭は「{担当者名}様」または「ご担当者様」で始める
・「突然のご連絡失礼いたします。株式会社HUMAN ADの小林と申します。」を必ず入れる
・相手の会社名と事業内容に必ず触れる
・料金は絶対に記載しない（「詳細は打ち合わせでご説明いたします」とする）
・候補日時を3つ入れる
・嘘や架空の実績は絶対に入れない
・丁寧だが堅すぎない、親しみやすいトーンで
・最後は「何卒よろしくお願いいたします。」で締める

【フォローの場合】
・フォロー1回目: 前回の連絡に軽く触れる。しつこくしない。
・フォロー2回目: 新しい情報や事例を提供する形で。
・フォロー3回目: 最後の連絡であることを伝え、今後も機会があればと締める。
"""

ユーザーメッセージとして以下を送る：
"""
【送信先情報】
会社名: {company_name}
担当者名: {contact_name}（空欄の場合は「ご担当者様」）
業種: {category}
エリア: {area_prefecture} {area_city}
Webサイト: {website_url}

【DM種別】{選択されたDM種別}
【候補日時】{date1}, {date2}, {date3}
【追加指示】{ユーザーの自由入力}

上記情報を踏まえて、最適なDMメッセージを1通作成してください。
"""

--- 5-5. AI生成エラー時のフォールバック ---
Claude APIがエラーを返した場合の対処：
1. タイムアウト（30秒）: 「生成に時間がかかっています。再試行してください」
2. 429エラー（レート制限）: 「APIが混み合っています。30秒後に再試行してください」
3. 500エラー（サーバーエラー）: 「一時的なエラーです。しばらく待って再試行してください」
4. その他のエラー: 「DM生成に失敗しました。手動で文面を入力してください」

全てのエラーで「手動入力に切り替える」ボタンを表示し、
テキストエリアに直接入力できるフォールバックUIを提供する。
AIが使えなくても営業は止まらないようにする。

--- 5-6. 文字数チェック ---
- 生成されたDMの文字数をリアルタイム表示
- 350〜550文字: 緑（OK）
- 350文字未満: 赤（短すぎ）
- 550文字超: 赤（長すぎ）
- 範囲外の場合は「再生成」ボタンで再度AIに生成させられる

--- 5-7. 送信フロー ---
AIでDM生成 → プレビュー・編集 → コピーボタン → FacebookのDM画面に貼り付け → 手動送信
※ 自動送信は行わない（Facebook規約遵守）

--- 5-8. Facebook DM画面への導線 ---
facebook_urlが保存されている場合：
  URL変換ロジック:
  1. https://www.facebook.com/{ページ名} → https://m.me/{ページ名} に変換
  2. https://www.facebook.com/{ページ名}/ → 末尾スラッシュ除去してから変換
  3. https://facebook.com/{ページ名} → www付きに正規化してから変換

  変換できないケース（フォールバック）:
  - profile.php?id=XXX 形式（個人ページ）→ m.me変換不可
  - /people/ を含むURL → m.me変換不可
  - /groups/ を含むURL → グループなのでDM不可
  上記の場合は元のfacebook_urlをそのまま「Facebookページを開く」リンクで表示し、
  ユーザーに手動でDMボタンを探してもらう。

  ボタンテキスト:
  - 変換成功時: 「FacebookでDMを開く」（m.meリンク）
  - 変換不可時: 「Facebookページを開く」（元URLリンク）

facebook_urlが空欄の場合：
  - 「Facebook未登録」と表示
  - 「Facebookで検索する」リンクを表示
    → https://www.facebook.com/search/pages/?q={会社名} に遷移

--- 5-9. 候補日時の管理 ---
現在（v1）: 手動入力（テキストフィールド × 3つ）
  - プレースホルダー例: 「来週火曜 14:00」「来週水曜 11:00」「来週木曜 15:00」

将来対応（v2で追加予定）:
  - Googleカレンダー連携（n8n経由で空き時間を自動取得）
  - 空き時間から候補を自動提案
  - ※現時点ではUIに「Googleカレンダー連携（準備中）」のグレーアウトボタンを
    配置しておくこと（将来の導線確保）

【6】ステータス管理・フォローアップ
================================================================

--- 6-1. ステータスフロー ---
未送信 → 初回DM送信済 → フォロー1回目 → フォロー2回目 → フォロー3回目（最終）→（手動でNG判断）
                ↓              ↓              ↓              ↓
              返信あり ←←←←←←←←←←←←←←←←←←←←←←←←←←←
                ↓
              商談中
                ↓
              成約

--- 6-2. フォロータイミング（全てJST基準で計算）---
- フォロー1回目: 初回DM送信から3日後
- フォロー2回目: フォロー1から4日後
- フォロー3回目（最終）: フォロー2から7日後
- next_follow_dateは送信時に自動計算してDBに保存
- 日付計算は全て Asia/Tokyo タイムゾーンで行うこと

--- 6-3. フォロー3回目後のNG判定 ---
フォロー3回目を送信後、返信がなくてもステータスは「フォロー3回目（最終）」のまま。
NGへの変更は手動で行う（自動NG化はしない）。
理由: 時間が経ってから返信が来る場合があるため、人間が判断する。

--- 6-4. フォローアラート ---
ダッシュボードに「今日のフォロー対象」セクションを表示
- next_follow_dateが今日（JST）以前のリードを自動抽出
- ワンクリックでDM生成画面に遷移（フォロー種別を自動選択）
- 件数バッジ表示（例: 「今日のフォロー対象: 12件」）

--- 6-5. フォローテンプレート自動選択 ---
- フォロー回数に応じてDM種別を自動推薦
  - 1回目: 種別D（リマインド型）
  - 2回目: 種別E（価値提供型）
  - 3回目: 種別F（ラストチャンス型）
